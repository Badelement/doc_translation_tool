- 若后续同一问题再次出现，应在原条目下补充，不重复开新条目。

  - 会让测试集持续处于失败状态，影响后续迭代判断
- 处理：
  - 已将 `tests/test_smoke.py` 的断言改为当前状态字符串

补充：

- 在后续推进 `T06`、`T07` 时，同类问题再次出现

- 这次处理不再只是同步字符串，而是把测试改成检查稳定结构：
  - `Document Translation Tool`
  - `Project root:`
  - `Status:`
  - `Next task:`
- 当前结论：
  - 该问题已从“具体字符串不同步”修正为“测试策略过脆”

### 问题 003：Qt 测试全部通过，但 Python 进程退出码异常

- 发现阶段：`T07` 回归测试
- 现象：
  - `pytest` 输出显示全部通过
  - 但 `python -m pytest -q` 的 `LASTEXITCODE` 异常，不是 `0`
- 原因：
  - GUI 测试里 `QApplication` 和窗口对象的清理方式不够稳定
  - 导致 Qt 测试进程退出阶段存在资源释放问题
- 影响范围：
  - 会让“测试看起来通过，但进程实际异常退出”
  - 这类问题如果不提前修，后面测试越多越难定位
- 处理：
  - 新增 `tests/conftest.py`，统一提供 session 级 `QApplication` fixture
  - 在窗口测试结束后显式执行 `close()` 和 `deleteLater()`
- 当前状态：已修复，当前测试结果为 `14 passed`，退出码 `0`

### 问题 004：语言检测初版对英文文本过于保守

- 发现阶段：`T10` 回归测试
- 现象：
  - 纯英文技术说明文本被判成 `mixed_or_unknown`
  - 方向切换提醒相关测试未触发预期警告
- 原因：
  - 初版英文判断阈值设置过高
  - 方向提醒测试样例使用了“语言与方向本来匹配”的场景
- 影响范围：
  - 会导致英文文件在 `英译中` / `中译英` 方向提示上表现不稳定
  - 影响语言检测的可用性，但不影响输入校验主链路
- 处理：
  - 将中英文启发式阈值从 `20` 下调到更适合技术文档短文本的 `6`
  - 修正方向切换测试场景，确保覆盖真正的不匹配情况
- 当前状态：已修复，相关测试通过

### 问题 005：Markdown 保护恢复时未保留原始末尾换行

- 发现阶段：`T12` 回归测试
- 现象：
  - 保护与恢复后的文本主体正确
  - 但原文末尾带换行时，恢复结果少了最后一个换行符
- 原因：
  - 解析器使用 `splitlines()` 后没有记录原文是否以换行结尾
  - 恢复阶段只按块用 `\n` 拼接，无法还原最终末尾换行状态
- 影响范围：
  - 会导致“恢复后文本和原文只差最后一个换行”
  - 后续如果不修，文件回填阶段会出现无意义 diff
- 处理：
  - 在 `MarkdownDocument` 和 `ProtectedMarkdownDocument` 中增加 `trailing_newline` 标记
  - 恢复时按原始状态补回末尾换行
- 当前状态：已修复，相关测试通过

### 问题 006：抽取提示词模块后出现循环导入

- 发现阶段：`T14` 回归测试
- 现象：
  - `pytest` 在收集 `llm` 相关测试时直接报 `ImportError`
  - 原因信息指向 `prompt.py` 与 `client.py` 的相互导入
- 原因：
  - `client.py` 导入 `prompt.py`
  - `prompt.py` 又反向导入 `TranslationItem`
  - 形成循环依赖
- 影响范围：
  - 阻塞 `llm` 模块导入
  - 相关测试全部无法运行
- 处理：
  - 将 `prompt.py` 中的具体类型依赖改为 `Protocol` 形式的结构约束
  - 避免 `prompt.py` 反向依赖 `client.py`
- 当前状态：已修复，测试恢复正常

### 问题 007：打包版配置文件方式容易让用户误用

- 发现阶段：`T24` 打包交付与首次用户试运行
- 现象：
  - 用户按打包目录中的示例文件理解为需要自行重命名 `.env.example`
  - 实际运行时容易出现 `LLM base_url is required` 一类错误
  - 同一个打包目录里同时存在示例文件和真实运行配置时，用户不容易判断程序到底读哪个
- 原因：
  - 早期打包产物偏向“开发者交付”思路，只强调模板文件，没有把“最终给用户直接改的配置文件”作为主入口
  - 使用说明虽然提到了 `.env`，但交付目录表达不够直接
- 影响范围：
  - 不影响源码逻辑本身
  - 会直接影响打包版首次上手成功率，属于高频易踩坑问题
- 处理：
  - 调整打包运行时配置策略，明确打包版从 `DocTranslationTool.exe` 同目录读取 `.env`
  - 打包目录直接放入可编辑的真实 `.env`，不再要求用户先复制或重命名模板
  - 更新 `使用指南.md`，明确“直接改 `.env`”而不是先找 `.env.example`
- 当前状态：已修复，打包目录已同步为“直接编辑 `.env`”的方案

### 问题 008：本地代理环境变量会干扰模型接口连接

- 发现阶段：真实接口联调
- 现象：
  - 同一组 `base_url`、`api_key` 和模型名，在工具里连接失败
  - 但直接按公司提供的 OpenAI 兼容格式请求时接口可用
- 原因：
  - `httpx` 默认会继承系统代理和环境变量
  - 在当前机器环境下，这会把原本应直连的内网接口带偏，导致连接异常
- 影响范围：
  - 影响所有通过 GUI 发起的真实模型请求
  - 会让用户误以为是模型名、Key 或接口地址配置错误
- 处理：
  - 在客户端初始化时显式设置 `trust_env=False`
  - 让程序忽略本机代理环境变量，按配置直连目标接口
- 当前状态：已修复，后续真实连通性验证已通过

### 问题 009：大批次翻译时模型可能破坏受保护占位符顺序

- 发现阶段：用户使用大文档实测
- 现象：
  - 翻译执行在后续批次失败，错误信息为：
    `Translated text changed protected placeholder tokens or their order.`
  - 日志可定位到具体批次和片段 ID，但按原逻辑会在重试耗尽后直接整体失败
- 原因：
  - 个别模型在较大批次下更容易改写占位符位置、顺序，或在占位符外残留中文
  - 结构校验虽然能拦住错误结果，但缺少更细粒度的自动恢复机制
- 影响范围：
  - 对小文档或简单批次影响较小
  - 对大文档、复杂表格或占位符较多的批次影响明显，属于真实使用中的关键稳定性问题
- 处理：
  - 保留原有批次级重试机制
  - 当失败原因属于结构类错误时，自动把失败批次递归拆成更小子批次继续重试
  - 日志中补充批次号、片段 ID 和 `[降批]` 提示，便于继续定位
- 当前状态：已修复，已补充对应测试并通过

### 问题 011：Windows 可能将未签名打包程序提示为潜在风险或潜在勒索程序

- 发现阶段：用户试运行打包版
- 现象：
  - 用户打开工具或尝试输出文件时，Windows 安全机制弹出“潜在勒索程序”或类似风险提示
- 原因：
  - 当前交付物是内部打包的未签名 Windows 可执行文件
  - 程序需要读取源文件并写入新文件，容易触发 Windows 的启发式拦截或受控文件夹访问提示
- 影响范围：
  - 不一定影响程序逻辑正确性
  - 但会影响试运行体验，甚至可能阻止输出文件写入
- 处理：
  - 在使用指南中补充该类提示的解释和排查建议
  - 建议优先使用普通测试目录输出，避免先写入桌面、文档等重点保护目录
  - 后续如果正式分发，建议做代码签名
- 当前状态：已记录，属于交付环境侧问题，代码层面暂不单独规避

### 问题 012：模型连接失败提示对 429 限流场景不够准确

- 发现阶段：用户试运行真实接口
- 现象：
  - 接口返回 `429 TOO MANY REQUESTS` 时，GUI 仍提示用户检查 `base_url`、`API key`、网络和代理
  - 实际上报错原因更接近平台限流、并发过高或额度限制
- 原因：
  - 早期客户端将所有 HTTP 错误都包装成通用连接失败
  - GUI 建议语句只按阶段给出固定提示，没有根据错误内容进一步细分
- 影响范围：
  - 不影响程序功能本身
  - 会让排查方向偏掉，增加用户误判成本
- 处理：
  - 在客户端中对 HTTP 状态错误做更明确的格式化
  - 对 `429` 单独输出限流提示
  - GUI 根据错误消息内容对 `429` 场景显示更准确的建议
- 当前状态：已修复，测试已补充

### 问题 013：服务端周额度耗尽会阻塞真实 API 联调

- 发现阶段：用户使用新打包版连接真实接口
- 现象：
  - 模型连接阶段返回 `429`
  - 返回体中包含 `code: throttling` 和 `message: week allocated quota exceeded.`
- 原因：
  - 请求已经成功到达接口提供方，但接口通道或 key 的周额度已用尽
  - 该问题位于服务端配额侧，不是本地配置、网络或程序逻辑错误
- 影响范围：
  - 会完全阻塞真实 API 路径的继续验证
  - 但不影响离线流程验证和非真实接口模式
- 处理：
  - 通过独立最小 HTTP 请求绕过 GUI 直接验证接口返回
  - 确认问题属于接口提供方配额限制
  - 提示改为引导用户更换 key 或联系接口管理员
- 当前状态：已确认，属于外部接口侧限制，不作为代码缺陷继续处理

### 问题 014：重打包时正式 `.env` 被模板覆盖，导致接口地址解析失败

- 发现阶段：新包重建后立即试运行
- 现象：
  - 打包版运行时报 `getaddrinfo failed`
  - 检查后发现 `dist\DocTranslationTool\.env` 被写成了模板里的占位地址 `https://example.internal/api/v1`
- 原因：
  - 打包脚本当时只会复制 `.env.example` 到发布目录
  - 项目根目录又没有真实 `.env`，导致正式包被模板值覆盖
- 影响范围：
  - 会让新打包版看起来“程序坏了”或“网络坏了”
  - 实际上只是运行配置被打错
- 处理：
  - 在项目根目录补入正式 `.env`
  - 修改打包脚本，优先复制真实 `.env`，仅在缺失时回退到 `.env.example`
  - 同步修正当前发布目录中的 `.env`
- 当前状态：已修复

### 问题 015：需要一种不依赖真实 API 额度的完整流程验证方式

  - 用打包目录实跑验证 `README.md` 到输出文件，确认 GUI 到输出文件的整条链路可用
- 当前状态：已修复，离线演示模式可用

  - 保留普通模板文件复制逻辑
  - 对 `使用指南.md` 改为使用 Unicode 码点拼接文件名，再单独复制
  - 重新 clean build，确认日志中出现 `Copied user guide`

### 问题 021：模型配置或术语表失败时，pipeline 清理阶段不够稳健

- 发现阶段：Mac 开发环境接手后代码复核
- 现象：
  - 当 `settings_loader()` 或 `client_factory()` 在客户端创建前失败时
  - `pipeline.execute()` 末尾的 `client.close()` 可能引用未赋值变量，覆盖原本应暴露的 `model_config` 错误
  - 当客户端已创建但术语表加载失败时，函数会在进入翻译主链路前直接抛错，导致客户端未被关闭
- 原因：
  - 客户端清理逻辑放在过晚的 `finally` 块中
  - 该 `finally` 默认假设客户端一定已经创建成功，且没有覆盖“客户端已创建但主流程尚未开始”的失败路径
- 影响范围：
  - 会让配置类错误暴露出错误的异常形态，增加排查成本
  - 也会在部分失败路径上留下不必要的资源清理缺口
- 处理：
  - 先将 `client` 初始化为 `None`
  - 将术语表加载一并纳入统一清理范围
  - 在 `finally` 中仅在客户端已成功创建时才执行关闭
  - 补充回归测试，覆盖“模型配置失败不被掩盖”和“术语表失败仍关闭客户端”两种场景
- 当前状态：已修复，相关回归测试已补充

### 问题 028：并行翻译时任务状态会直接跳到最后一批，进度条也不够贴近真实推进

- 发现阶段：发布 `v0.4.0` 后继续验证大文档翻译体验时
- 现象：
  - 一开始翻译就显示类似 `正在处理第 98/98 批`
  - 任务状态看起来像已经跑到最后一批，但日志和真实执行节奏并不匹配
  - 进度条在多批次场景下推进不明显，前几批完成后用户体感上仍像“没动”
- 原因：
  - 并行模式下，旧逻辑会在开始时把所有批次的 `on_batch_started` 事件一次性抛给界面
  - 最后一个开始事件会覆盖前面的状态，导致 GUI 直接停在“最后一批”
  - 翻译阶段进度算法对大批次场景不够敏感，且早期使用向下取整，导致前几批完成时进度变化过小
- 影响范围：
  - 不影响翻译结果正确性
  - 会明显影响用户对程序是否卡住、是否真正推进中的判断
- 处理：
  - 将并行调度改为只维持当前并发窗口，不再一开始提交并广播所有批次的开始状态
  - 只有真正进入活动 worker 的批次才会触发 `on_batch_started`
  - 调整翻译阶段进度区间和计算方式，让批次完成后进度条推进更明显
  - 补充 task service 与 pipeline 回归测试，覆盖“不要一开始跳到最后一批”和“翻译阶段进度值更新”这两个场景
- 当前状态：已修复，当前全量测试结果为 `129 passed`

# 文档翻译工具使用指南

当前发布版本：`v0.4.6`

## 1. 这是什么

这是一个用来翻译单个 Markdown / DITA 文档，或批量翻译一个目录中支持文档的桌面工具。

它更适合这类场景：

- 你要翻译的是一个 `.md` / `.dita` 文件，或者一个目录里的多份支持文档
- 你希望尽量保留 Markdown 结构或 DITA XML 文本结构
- 你不想覆盖原文件，而是生成一个新文件
- 你处理的是技术文档、表格较多的文档，或者夹杂路径、代码、标签的文档

支持的方向：

- `中译英`
- `英译中`

## 2. 最快上手

### Windows 用户

1. 把整个 `DocTranslationTool-win64.zip` 解压到一个普通目录
2. 打开解压后的运行目录；你应该直接看到 `DocTranslationTool.exe`、`_internal\` 和 `.env`
3. 如果你是在源码目录里使用本地构建结果，对应目录是 `dist\DocTranslationTool\`
4. 打开 `.env`，先只改这 3 项：

```env
DOC_TRANS_BASE_URL=你的接口地址
DOC_TRANS_API_KEY=你的API Key
DOC_TRANS_MODEL=你要用的模型名
```

5. 保存 `.env`
6. 双击运行 `DocTranslationTool.exe`

### macOS 用户

1. 把整个 `DocTranslationTool-macos.zip` 解压到一个普通目录
2. 打开解压后的目录；你应该看到 `DocTranslationTool.app` 和 `.env`
3. 打开 `.env`，先只改这 3 项：

```env
DOC_TRANS_BASE_URL=你的接口地址
DOC_TRANS_API_KEY=你的API Key
DOC_TRANS_MODEL=你要用的模型名
```

4. 保存 `.env`
5. 双击 `DocTranslationTool.app` 启动应用
6. 如果 macOS 提示安全警告，右键点击应用，选择”打开”，然后在弹出对话框中再次点击”打开”

### 从源码运行（所有平台）

```bash
# 安装依赖
python3 -m pip install -e .

# 复制并编辑配置
cp .env.example .env
# 编辑 .env 填入你的 API 配置

# 运行应用
python3 app.py
```

### 使用应用

1. 先看左下角”模型状态”
2. 在”单文件”或”目录批量”里选择任务模式
3. 选择一个 `.md` / `.dita` 文件，或者选择源目录，并选择输出目录
4. 如有需要，手动切换翻译方向
5. 点击”开始翻译”

大多数人第一次使用，只需要这 3 个配置。

## 3. 你真正需要改什么

对大多数用户来说：

- 只需要改 `.env`
- 不需要改 `settings.json`
- 不需要改代码
- 不需要重新打包

最重要的 3 项是：

- `DOC_TRANS_BASE_URL`：你的模型接口地址
- `DOC_TRANS_API_KEY`：你的接口密钥
- `DOC_TRANS_MODEL`：你的模型名

如果这 3 项已经填好，其他项可以先保持默认。

## 4. 这些配置文件分别是干什么的

你通常会看到这几个文件：

- `.env.example`：`.env` 的模板文件
- `settings.example.json`：`settings.json` 的模板文件
- `glossary.example.json`：`glossary.json` 的模板文件
- `.env`：最推荐日常修改的运行配置
- `settings.json`：可选的高级 JSON 配置
- `glossary.json`：真正会在翻译时自动加载的术语表

最实用的理解方式是：

- `.env` 管“接口地址、API Key、模型名”这类你最常改的东西
- `settings.json` 管“批大小、并发、重试、超时、front matter 字段”这类偏高级的默认项
- `glossary.json` 管“固定术语怎么翻，或者哪些词保持原样”

如果你只是第一次跑通工具：

- 先改 `.env`
- 先不要动 `settings.json`
- 需要固定术语时，再去配 `glossary.json` 或直接用界面里的“术语配置”

## 5. `settings.json` 和 `.env` 是什么关系

程序读取配置时，优先级是这样的：

1. 运行时环境变量
2. `.env`
3. `settings.json`
4. 代码默认值

这意味着：

- 如果同一个配置同时写在 `.env` 和 `settings.json` 里，以 `.env` 为准
- 如果你只在 `settings.json` 里写了某个字段，而 `.env` 没写，那程序会用 `settings.json`
- 如果两边都没写，就回退到程序默认值

建议你按下面分工来用：

- `.env`：放 `DOC_TRANS_BASE_URL`、`DOC_TRANS_API_KEY`、`DOC_TRANS_MODEL`
- `settings.json`：放 `batch_size`、`parallel_batches`、`max_retries`、`timeout`、`max_tokens` 这类高级参数

简单说：

- 想快速改模型接入参数，用 `.env`
- 想固定一套高级默认参数，用 `settings.json`
- 两边不要重复写同一个字段，除非你明确知道自己要让 `.env` 覆盖 `settings.json`

## 6. `settings.example.json` 怎么用

`settings.example.json` 只是模板，不会被程序自动读取。

正确用法是：

1. 把 `settings.example.json` 复制一份并改名成 `settings.json`
2. 再按需要修改里面的字段

它更适合放这些内容：

- `llm.batch_size`
- `llm.parallel_batches`
- `llm.max_retries`
- `llm.timeout`
- `llm.connect_timeout`
- `llm.read_timeout`
- `llm.max_tokens`
- `markdown.front_matter_translatable_fields`

如果你只是普通使用者，可以完全不创建 `settings.json`。

如果你想看每个参数更具体的解释，例如：

- 它到底控制什么
- 默认值是多少
- 对应哪个 `.env` 变量
- 什么时候应该调大或调小

请优先查看当前目录里的 `settings参数说明.md`。
如果你拿到的是精简交付目录，当前目录没有这份扩展说明，再回源码目录查看即可。

## 7. `glossary.example.json` 和术语表怎么用

`glossary.example.json` 也是模板，不会直接生效。真正生效的是 `glossary.json`。

你有两种使用方式：

1. 手动方式

- 把 `glossary.example.json` 复制成 `glossary.json`
- 按下面这种格式填写：

```json
[
  {
    "source": "远程音效",
    "target": "remote sound effect"
  },
  {
    "source": "Android13",
    "target": "Android13"
  }
]
```

2. 界面方式

- 打开主界面
- 点击“术语配置”
- 在表格里新增、删除术语
- 保存后程序会自动写入 `glossary.json`

术语表的作用是：

- 遇到 `source` 时，优先按 `target` 去翻
- 如果你想“保持原样”，就把 `source` 和 `target` 设成一样
- 比如 `Android13 -> Android13`，意思就是尽量不要翻动这个词

更适合放进术语表的内容：

- 产品名
- 模块名
- 平台名
- 内部固定说法
- 不希望模型自由发挥的技术词

不建议把大量普通词都塞进去，不然术语表噪音会变大。

## 8. `.env` 怎么填

### 8.1 OpenAI 兼容接口

如果你的平台是 OpenAI 兼容接口，通常这样填：

```env
DOC_TRANS_PROVIDER=openai_compatible
DOC_TRANS_API_FORMAT=openai
DOC_TRANS_BASE_URL=https://your-service.example/v1
DOC_TRANS_API_KEY=your_api_key_here
DOC_TRANS_MODEL=your_model_name
```

### 8.2 Anthropic 兼容接口

如果你的平台是 Anthropic 兼容接口，最简单的方式是这样填：

```env
DOC_TRANS_PROVIDER=openai_compatible
DOC_TRANS_API_FORMAT=anthropic
DOC_TRANS_ANTHROPIC_VERSION=2023-06-01
DOC_TRANS_BASE_URL=https://your-service.example
DOC_TRANS_API_KEY=your_api_key_here
DOC_TRANS_MODEL=your_model_name
```

这里最容易填错的是：

- `DOC_TRANS_API_FORMAT` 要改成 `anthropic`
- `DOC_TRANS_BASE_URL` 一般填服务根地址，不要自己再手动拼 `/messages`
- `DOC_TRANS_ANTHROPIC_VERSION` 一般保持默认 `2023-06-01`，除非你的服务商明确要求别的值

如果你们平台要求把 `DOC_TRANS_PROVIDER` 写成 `anthropic_compatible`，也可以；但只要当前服务是 Anthropic 兼容格式，关键还是 `DOC_TRANS_API_FORMAT=anthropic`。

## 9. 正常使用流程

1. 打开 `DocTranslationTool.exe`
2. 确认左下角模型状态不是“配置不完整”
3. 选择任务模式：
4. `单文件`：适合一次翻译一个文档
5. `目录批量`：适合一次处理同一目录里的多个支持文档
6. 选择要翻译的 `.md` / `.dita` 文件，或者选择源目录
7. 选择输出目录
8. 确认翻译方向
9. 点击“开始翻译”
10. 等待任务完成
11. 到输出目录查看结果文件

输出文件不会覆盖原文件。

默认命名规则：

- 中译英：`原文件名_en.原扩展名`
- 英译中：`原文件名_zh.原扩展名`

第一次使用建议先拿一个小一点的文档试一次，确认接口、模型和输出都正常，再去跑大文档。

目录批量模式还要注意这几点：

- 当前会自动扫描你选择的源目录
- 只有支持的 `.md` / `.dita` 文件会进入翻译队列
- 所有扫描到的支持文件共享同一个输出目录
- 程序会按文件分别自动识别语言，并自动决定每个文件的翻译方向
- 界面上的方向选择在目录批量里只作为“未确定文件”的默认方向
- 程序会按文件名顺序逐个执行，并在最后给出成功 / 失败 / 跳过汇总
- 如果目录里已经有工具之前生成的 `*_en` / `*_zh` 文件，程序会尽量自动跳过，避免二次翻译

## 10. 界面状态怎么看

你通常会看到这些状态：

- `模型状态：配置不完整`：关键配置没填全
- `模型状态：配置已加载，待检查连通性`：本地配置已经读到，但还没真正发请求
- `模型状态：正在检查接口连通性`：程序正在验证接口是否可用
- `翻译中：正在处理第 x/N 批`：当前有批次在执行
- `翻译中：已完成批次 x/N`：已经有批次返回结果，进度会继续推进

简单理解就是：

- 程序启动时只检查“配置像不像能用”
- 真正访问模型接口，是在你点了“开始翻译”之后

## 11. 大而复杂的文档怎么配

如果你翻译的是很大、很复杂、很技术化的 Markdown 文档，或者结构比较规整的 DITA 文档，建议按“成功率 > 质量 > 速度”的思路来配。

建议从这组参数开始：

```env
DOC_TRANS_BATCH_SIZE=4
DOC_TRANS_PARALLEL_BATCHES=1
DOC_TRANS_VALIDATION_MODE=lenient
```

说明：
- `VALIDATION_MODE=lenient` 会放宽对技术术语和占位符顺序的检查
- 适合包含大量英文专业术语、代码、路径的技术文档

如果跑得比较稳，再慢慢加到：

```env
DOC_TRANS_PARALLEL_BATCHES=2
```

实用建议：

- 文档越复杂，越不要一开始就把并发调很高
- 表格很多、路径很多、标签很多的文档，更适合保守配置
- 如果中途失败，尽量用同一个输出目录重试，这样可以复用已缓存的已翻译片段
- 如果接口容易限流，优先降低 `DOC_TRANS_PARALLEL_BATCHES`

当前版本已经对一些常见复杂场景做了保护，例如：

- 超长表格行更安全地切分
- HTML/XML 标签保护
- 文件名、路径、技术常量保护
- 命中 `429` 时自动降并发重试
- 中断后 checkpoint / resume
- 自动识别技术文档并应用宽松验证模式
- 实时段落级进度反馈和动态耗时预估
- 重试原因透明化，便于排查问题

如果你翻的是 `.dita` 文件，还要额外注意：

- 当前更适合常见 topic / task 类 DITA 内容
- 输出会保留 `.dita` 后缀，不会改成 `.md`
- 建议翻译后抽查标题、表格单元格、`xref` 周边文本和说明性段落

## 12. 常见问题先看这里

### 12.1 显示“配置不完整”

优先检查这 3 项有没有填：

- `DOC_TRANS_BASE_URL`
- `DOC_TRANS_API_KEY`
- `DOC_TRANS_MODEL`

### 12.2 我改了 `.env`，为什么没生效

优先检查：

- 你改的是不是 `DocTranslationTool.exe` 同目录下的 `.env`
- 有没有把程序放在 zip 压缩包里直接运行
- 有没有只拷贝 `exe`，却没把整个目录一起带上

### 12.3 Anthropic 兼容接口还是连不上

优先检查：

- `DOC_TRANS_API_FORMAT` 是否已经设为 `anthropic`
- `DOC_TRANS_BASE_URL` 是否多写了 `/messages`
- `DOC_TRANS_ANTHROPIC_VERSION` 是否被改成了服务端不接受的值

### 12.4 大文档翻译慢，或者中途报限流

先不要急着把参数调大，反而应该先调小：

- 把 `DOC_TRANS_PARALLEL_BATCHES` 调低
- 必要时把 `DOC_TRANS_BATCH_SIZE` 也调低
- 保留同一个输出目录重试

### 12.5 目录批量里同时有中文文件和英文文件怎么办

当前目录批量会按文件分别自动识别方向。

这意味着：

- 中文文件会优先按 `中译英` 处理
- 英文文件会优先按 `英译中` 处理
- 如果某个文件的语言判断不够明确，程序会退回到你界面上当前选择的默认方向

更稳妥的做法仍然是：

- 对语言非常明确的大批中文文件，可以把默认方向设成 `中译英`
- 对语言非常明确的大批英文文件，可以把默认方向设成 `英译中`
- 如果目录里混入很短、正文很少、标签很多的文件，开始前先看一下程序给出的汇总提示

### 12.6 当前主界面的目录批量怎么理解

当前主界面的目录批量，就是以“源目录”为输入的批量任务：

- 你选择一个目录作为输入
- 程序会自动扫描目录中的支持文件
- 不支持的文件会被跳过
- 疑似工具之前生成的输出文件会被自动跳过
- 扫描到的支持文件会共享同一个输出目录
- 每个文件会单独识别语言并决定方向

## 13. 如果只想演示流程，不想占用真实额度

可以临时切到离线演示模式：

```env
DOC_TRANS_PROVIDER=mock
DOC_TRANS_MODEL=mock-demo
```

这个模式下：

- 不会调用真实 API
- 可以演示 GUI、分段、回填和输出流程
- 输出内容只是模拟结果，不适合正式使用

演示完记得切回真实配置。

- 若后续同一问题再次出现，应在原条目下补充，不重复开新条目。

补充：

  - `Project root:`

  - `Next task:`

### 问题 003：Qt 测试全部通过，但 Python 进程退出码异常

- 发现阶段：`T07` 回归测试
- 现象：
  - `pytest` 输出显示全部通过
  - 但 `python -m pytest -q` 的 `LASTEXITCODE` 异常，不是 `0`
- 原因：
  - GUI 测试里 `QApplication` 和窗口对象的清理方式不够稳定
  - 导致 Qt 测试进程退出阶段存在资源释放问题
- 影响范围：
  - 会让“测试看起来通过，但进程实际异常退出”
  - 这类问题如果不提前修，后面测试越多越难定位
- 处理：
  - 新增 `tests/conftest.py`，统一提供 session 级 `QApplication` fixture
  - 在窗口测试结束后显式执行 `close()` 和 `deleteLater()`
- 当前状态：已修复，当前测试结果为 `14 passed`，退出码 `0`

### 问题 004：语言检测初版对英文文本过于保守

- 发现阶段：`T10` 回归测试
- 现象：
  - 纯英文技术说明文本被判成 `mixed_or_unknown`
  - 方向切换提醒相关测试未触发预期警告
- 原因：
  - 初版英文判断阈值设置过高
  - 方向提醒测试样例使用了“语言与方向本来匹配”的场景
- 影响范围：
  - 会导致英文文件在 `英译中` / `中译英` 方向提示上表现不稳定
  - 影响语言检测的可用性，但不影响输入校验主链路
- 处理：
  - 将中英文启发式阈值从 `20` 下调到更适合技术文档短文本的 `6`
  - 修正方向切换测试场景，确保覆盖真正的不匹配情况
- 当前状态：已修复，相关测试通过

### 问题 005：Markdown 保护恢复时未保留原始末尾换行

- 发现阶段：`T12` 回归测试
- 现象：
  - 保护与恢复后的文本主体正确
  - 但原文末尾带换行时，恢复结果少了最后一个换行符
- 原因：
  - 解析器使用 `splitlines()` 后没有记录原文是否以换行结尾
  - 恢复阶段只按块用 `\n` 拼接，无法还原最终末尾换行状态
- 影响范围：
  - 会导致“恢复后文本和原文只差最后一个换行”
  - 后续如果不修，文件回填阶段会出现无意义 diff
- 处理：
  - 在 `MarkdownDocument` 和 `ProtectedMarkdownDocument` 中增加 `trailing_newline` 标记
  - 恢复时按原始状态补回末尾换行
- 当前状态：已修复，相关测试通过

### 问题 006：抽取提示词模块后出现循环导入

- 发现阶段：`T14` 回归测试
- 现象：
  - `pytest` 在收集 `llm` 相关测试时直接报 `ImportError`
  - 原因信息指向 `prompt.py` 与 `client.py` 的相互导入
- 原因：
  - `client.py` 导入 `prompt.py`
  - `prompt.py` 又反向导入 `TranslationItem`
  - 形成循环依赖
- 影响范围：
  - 阻塞 `llm` 模块导入
  - 相关测试全部无法运行
- 处理：
  - 将 `prompt.py` 中的具体类型依赖改为 `Protocol` 形式的结构约束
  - 避免 `prompt.py` 反向依赖 `client.py`
- 当前状态：已修复，测试恢复正常

### 问题 007：打包版配置文件方式容易让用户误用

- 发现阶段：`T24` 打包交付与首次用户试运行
- 现象：
  - 用户按打包目录中的示例文件理解为需要自行重命名 `.env.example`
  - 实际运行时容易出现 `LLM base_url is required` 一类错误
  - 同一个打包目录里同时存在示例文件和真实运行配置时，用户不容易判断程序到底读哪个
- 原因：
  - 早期打包产物偏向“开发者交付”思路，只强调模板文件，没有把“最终给用户直接改的配置文件”作为主入口
  - 使用说明虽然提到了 `.env`，但交付目录表达不够直接
- 影响范围：
  - 不影响源码逻辑本身
  - 会直接影响打包版首次上手成功率，属于高频易踩坑问题
- 处理：
  - 调整打包运行时配置策略，明确打包版从 `DocTranslationTool.exe` 同目录读取 `.env`
  - 打包目录直接放入可编辑的真实 `.env`，不再要求用户先复制或重命名模板
  - 更新 `使用指南.md`，明确“直接改 `.env`”而不是先找 `.env.example`
- 当前状态：已修复，打包目录已同步为“直接编辑 `.env`”的方案

### 问题 008：本地代理环境变量会干扰模型接口连接

- 发现阶段：真实接口联调
- 现象：
  - 同一组 `base_url`、`api_key` 和模型名，在工具里连接失败
  - 但直接按公司提供的 OpenAI 兼容格式请求时接口可用
- 原因：
  - `httpx` 默认会继承系统代理和环境变量
  - 在当前机器环境下，这会把原本应直连的内网接口带偏，导致连接异常
- 影响范围：
  - 影响所有通过 GUI 发起的真实模型请求
  - 会让用户误以为是模型名、Key 或接口地址配置错误
- 处理：
  - 在客户端初始化时显式设置 `trust_env=False`
  - 让程序忽略本机代理环境变量，按配置直连目标接口
- 当前状态：已修复，后续真实连通性验证已通过

### 问题 009：大批次翻译时模型可能破坏受保护占位符顺序

- 发现阶段：用户使用大文档实测
- 现象：
  - 翻译执行在后续批次失败，错误信息为：
    `Translated text changed protected placeholder tokens or their order.`
  - 日志可定位到具体批次和片段 ID，但按原逻辑会在重试耗尽后直接整体失败
- 原因：
  - 个别模型在较大批次下更容易改写占位符位置、顺序，或在占位符外残留中文
  - 结构校验虽然能拦住错误结果，但缺少更细粒度的自动恢复机制
- 影响范围：
  - 对小文档或简单批次影响较小
  - 对大文档、复杂表格或占位符较多的批次影响明显，属于真实使用中的关键稳定性问题
- 处理：
  - 保留原有批次级重试机制
  - 当失败原因属于结构类错误时，自动把失败批次递归拆成更小子批次继续重试
  - 日志中补充批次号、片段 ID 和 `[降批]` 提示，便于继续定位
- 当前状态：已修复，已补充对应测试并通过

### 问题 011：Windows 可能将未签名打包程序提示为潜在风险或潜在勒索程序

- 发现阶段：用户试运行打包版
- 现象：
  - 用户打开工具或尝试输出文件时，Windows 安全机制弹出“潜在勒索程序”或类似风险提示

  - 当前交付物是内部打包的未签名 Windows 可执行文件
  - 程序需要读取源文件并写入新文件，容易触发 Windows 的启发式拦截或受控文件夹访问提示
- 影响范围：
  - 不一定影响程序逻辑正确性
  - 但会影响试运行体验，甚至可能阻止输出文件写入
- 处理：
  - 在使用指南中补充该类提示的解释和排查建议
  - 建议优先使用普通测试目录输出，避免先写入桌面、文档等重点保护目录
  - 后续如果正式分发，建议做代码签名

### 问题 012：模型连接失败提示对 429 限流场景不够准确

- 发现阶段：用户试运行真实接口
- 现象：
  - 接口返回 `429 TOO MANY REQUESTS` 时，GUI 仍提示用户检查 `base_url`、`API key`、网络和代理
  - 实际上报错原因更接近平台限流、并发过高或额度限制
- 原因：
  - 早期客户端将所有 HTTP 错误都包装成通用连接失败
  - GUI 建议语句只按阶段给出固定提示，没有根据错误内容进一步细分
- 影响范围：
  - 不影响程序功能本身
  - 会让排查方向偏掉，增加用户误判成本
- 处理：
  - 在客户端中对 HTTP 状态错误做更明确的格式化
  - 对 `429` 单独输出限流提示
  - GUI 根据错误消息内容对 `429` 场景显示更准确的建议
- 当前状态：已修复，测试已补充

### 问题 013：服务端周额度耗尽会阻塞真实 API 联调

- 发现阶段：用户使用新打包版连接真实接口
- 现象：
  - 模型连接阶段返回 `429`
  - 返回体中包含 `code: throttling` 和 `message: week allocated quota exceeded.`
- 原因：
  - 请求已经成功到达接口提供方，但接口通道或 key 的周额度已用尽
  - 该问题位于服务端配额侧，不是本地配置、网络或程序逻辑错误
- 影响范围：
  - 会完全阻塞真实 API 路径的继续验证
  - 但不影响离线流程验证和非真实接口模式
- 处理：
  - 通过独立最小 HTTP 请求绕过 GUI 直接验证接口返回
  - 确认问题属于接口提供方配额限制
  - 提示改为引导用户更换 key 或联系接口管理员
- 当前状态：已确认，属于外部接口侧限制，不作为代码缺陷继续处理

### 问题 014：重打包时正式 `.env` 被模板覆盖，导致接口地址解析失败

- 发现阶段：新包重建后立即试运行
- 现象：
  - 打包版运行时报 `getaddrinfo failed`
  - 检查后发现 `dist\DocTranslationTool\.env` 被写成了模板里的占位地址 `https://example.internal/api/v1`
- 原因：
  - 打包脚本当时只会复制 `.env.example` 到发布目录
  - 项目根目录又没有真实 `.env`，导致正式包被模板值覆盖
- 影响范围：
  - 会让新打包版看起来“程序坏了”或“网络坏了”
  - 实际上只是运行配置被打错
- 处理：
  - 在项目根目录补入正式 `.env`
  - 修改打包脚本，优先复制真实 `.env`，仅在缺失时回退到 `.env.example`
  - 同步修正当前发布目录中的 `.env`
- 当前状态：已修复

### 问题 015：需要一种不依赖真实 API 额度的完整流程验证方式

- 补充结论：
  - 这一步优先提升的是复杂技术文档的翻译质量和技术字面量稳定性
  - 仍然是类别化规则，不是针对某个文件名做特判

### 问题 033：大文档在高并发批量翻译时容易因 429/限流中途失败，用户必须手动降低并发重跑

- 发现阶段：围绕“翻译成功率优先”重新评估运行时策略时
- 现象：
  - 大文档会被切成上百个 batch，固定并发在某些模型服务上容易触发 `429 Too Many Requests`
  - 旧实现能够识别 429 并提示用户，但会直接把任务失败抛出
  - 用户需要手动去改 `parallel_batches` 后再重跑，体验较差
- 原因：
  - 旧逻辑只有“批次重试”，没有“命中限流后自动降载”的能力
  - 当重试仍沿用原并发度时，很容易继续被限流
- 影响范围：
  - 主要影响复杂大文档的翻译成功率和速度稳定性
  - 尤其在第三方模型服务限流策略较严格时更明显
- 处理：
  - 重构 `TranslationTaskService` 内部并发调度逻辑，把顺序执行和并发执行分别抽成清晰 helper
  - 只对 `429` / `rate limited` / `throttling` 这类明确限流错误触发自适应降并发
  - 命中限流后，对未完成批次降并发继续处理，而不是直接让整个任务失败
  - 新增针对“并发中命中 429 后自动降并发继续完成”的回归测试
- 当前状态：已修复
- 补充结论：
  - 该修复优先提升成功率，其次改善大文档吞吐稳定性
  - 当前版本未加入等待退避时间，先保持最小可验证实现，避免复杂度失控

### 问题 034：大文档中途失败后必须从头重翻，已完成批次无法复用，导致恢复成本过高

- 发现阶段：继续围绕复杂技术文档的真实使用体验优化时
- 现象：
  - 像 `PMIC`、`MIPI_CSI` 这类大文档会被切成上千个 segment、上百个 batch
  - 旧实现一旦中途失败，下次重跑仍然从第一个 batch 重新开始
  - 如果失败发生在靠后批次，时间浪费非常明显
- 原因：
  - 原有 pipeline 只保存最终输出，不保存中间的 segment 翻译结果
  - task service 没有“跳过已完成片段继续翻剩余片段”的接口
- 影响范围：
  - 直接影响大文档场景下的恢复效率、重试成本和整体用户体验
  - 对复杂文档尤其明显，因为翻译耗时和失败概率都更高
- 处理：
  - 新增独立的 `translation_cache.py` 模块，负责缓存路径、文档指纹、缓存读写和清理
  - 为 `TranslationTaskService.translate_segmented_document()` 增加 `existing_translations` 与 `on_batch_translated` 接口
  - pipeline 在翻译前自动加载缓存、翻译中逐批持久化缓存、成功写出输出文件后自动清理缓存
  - 使用文档指纹保证“源文档变了就自动让旧缓存失效”
  - 新增 task service、缓存模块和 pipeline 续跑闭环测试
- 当前状态：已修复
- 补充结论：
  - 当前实现属于“轻量续跑”，不是完整任务数据库，但已经能显著降低大文档中途失败后的重跑成本
  - 这一步优先提升的是成功率体验和大文档恢复效率，而不是翻译质量本身

### 问题 035：单片段中的多个受保护占位符被模型自然改写顺序时，原有回退路径仍可能整体失败

- 发现阶段：用户使用真实大文档和打包版说明文档实测时
- 现象：
  - `Allwinner_Developer_Guide_Cedar_Sound_Effect_Tuning_zh.md` 中某些单片段包含多个受保护占位符时，会报错：`Translated text changed protected placeholder tokens or their order.`
  - `DocTranslationTool-win64\README.md` 这类较短说明文档中，也可能在单片段上复现同类问题
  - 旧逻辑虽然支持“失败批次拆小重试”，但当失败批次本身只剩 `1` 个 segment 时，就没有更细粒度的自动恢复路径
- 原因：
  - 某些模型会对“包含多个受保护字面量的关系句”做自然语序调整
  - 结构校验要求占位符顺序完全一致，因此会把这种结果判定为失败
  - 但对单片段来说，继续做普通批次级重试几乎没有额外收益
- 影响范围：
  - 主要影响复杂技术文档和含多个文件名、路径、技术常量占位符的说明句
  - 会把原本可恢复的局部问题升级成整批甚至整文档失败
- 处理：
  - 在 `TranslationTaskService` 中新增“单片段占位符保序救援”路径
  - 仅当失败批次长度为 `1`、错误明确属于占位符顺序校验失败、且片段内至少有 `2` 个占位符时触发
  - 按占位符边界把原句拆成更小 chunk，分别翻译后再按原顺序拼回
  - 补充两占位符和三占位符的回归测试，覆盖真实样本模式
- 当前状态：已修复
- 补充结论：
  - 该优化是“在原有严格校验前提下增加更小粒度的恢复路径”，不是放松占位符保护规则
  - 优先提升的是复杂文档场景下的翻译成功率

### 问题 048：DITA 重建使用 ElementTree 默认序列化时，会丢失树内注释/处理指令并改写命名空间前缀

- 发现阶段：代码审查与本地最小复现
- 现象：
  - 某些 DITA/XML 输入包含树内注释、processing instruction 或显式命名空间前缀时，重建后的输出会发生结构漂移
  - 注释与处理指令可能在 round-trip 后消失
  - 类似 `xmlns:x="..."` / `x:role="..."` 的前缀可能被改写成 `ns0`
- 原因：
  - 早期实现使用 `xml.etree.ElementTree.fromstring()/tostring()` 的默认行为
  - 默认 `TreeBuilder` 不保留 comment / PI 节点，且序列化时不会自动保留原始前缀映射
- 影响范围：
  - 影响需要尽量无损保留 XML 结构信息的 DITA 文档
  - 虽不一定影响最终可解析性，但会制造不必要的 diff，部分下游工具链也可能依赖这些节点或前缀约定
- 处理：
  - 为安全 XML parser 补充 `TreeBuilder(insert_comments=True, insert_pis=True)`，保留树内 comment / PI
  - 在 prepare 阶段提取源文档命名空间映射，在 rebuild 前重新注册命名空间前缀
  - 补充回归测试，覆盖树内注释、processing instruction 和命名空间前缀保持场景
- 当前状态：已修复
- 补充结论：
  - 这次修复的目标是让 DITA 重建更接近“结构稳定的 round-trip”，而不是只关注可翻译文本节点
  - 已随 `v0.4.5` 一并重建 Windows 发布包，当前发布目录与 zip 产物均已同步

### 问题 049：单个英文技术段在保留少量稳定英文术语时，`en_to_zh` 残留英文校验可能把整段任务误判为失败

  - 补充回归测试，覆盖与 `CLAUDE.md` 同类的两行英文技术说明场景
- 当前状态：已修复
- 补充结论：
  - 这次修复不是放松整体质量要求，而是把单段恢复路径从“只有整体重试”扩展到“可按更小语义块重试”
  - 该热修已随 `v0.4.6` 重新打包 Windows 发布产物

### 问题 050：批量翻译按文件顺序复用单文件流程时，每个文件都会重复创建 client / pipeline 并重复做连接检查，导致固定开销被放大

- 发现阶段：围绕新增轻量 benchmark 脚本对 batch 模式做性能基线测量
- 现象：
  - 目录批量翻译虽然已经按顺序复用单文件执行链路，但每处理一个文件都会重新构造一套 pipeline 运行时资源
  - `benchmark_translation.py` 的 batch 指标显示，4 文件样本下 `client_creations=4`、`connection_checks=4`
  - 在 mock provider 且人为增加固定延迟后，这类重复初始化会明显拉高总耗时，而 `translate_batch_calls` 与 `translated_items_sent` 并没有变化
- 原因：
  - `BatchTranslationService` 早期实现是在循环内部逐文件调用 `pipeline_factory()`，天然导致 pipeline、client 和 glossary 装载都按文件重复发生
  - `DocumentTranslationPipeline.execute()` 原本以“单文件独立执行”作为资源生命周期边界，默认每次都创建并关闭 client，并重新做 `check_connection`
  - 单文件语义本身没有问题，但直接平移到 batch 场景会把本可批量摊销的固定开销重复支付多次
- 影响范围：
  - 主要影响 batch 模式的总耗时、平均单文件耗时，以及真实 provider 下的连接准备开销
  - 不直接改变翻译质量或批次内吞吐，但会放大目录内文件较多时的固定成本
- 处理：
  - 在 `pipeline.py` 中新增轻量的 `TranslationPipelineRuntimeSession`，批量会话内懒加载并持有 `settings`、`glossary`、`client`
  - 将 `check_connection` 的成功结果缓存到这次批量会话中，使同一批量任务内后续文件不再重复探活
  - 保持单文件路径不变：未传入 runtime session 时，pipeline 仍按原语义自行创建并关闭资源
  - 将 `BatchTranslationService` 调整为一次批量任务只创建一个 pipeline，并在整个批量循环中复用同一个 runtime session，结束后统一关闭
  - 补充 pipeline 与 batch service 回归测试，验证“批量内只建一次 session / 只做一次连接检查 / 仍能在失败路径正确清理”
  - 使用现有 `scripts/benchmark_translation.py` 做前后对比，不引入额外的复杂 benchmark 基建
- 当前状态：已修复并完成 benchmark 对比
- 补充结论：
  - 同一组 4 文件样本、`--provider mock --mock-delay-ms 50` 条件下，改动前 `total_elapsed_seconds=0.573s`、`avg_file_seconds=0.143s`、`client_creations=4`、`connection_checks=4`
  - 改动后同条件结果为 `total_elapsed_seconds=0.395s`、`avg_file_seconds=0.099s`、`client_creations=1`、`connection_checks=1`
  - `translate_batch_calls=5`、`translated_items_sent=31` 在前后对比中保持不变，说明这次优化收缩的是重复初始化成本，而不是改变翻译工作量本身

  - 用户最终更希望目录批量做到三件事：自动扫描支持文件、按文件逐个翻译、优先复用单文件已经成熟的方向识别能力
- 原因：
  - 多文件入口虽然贴近“显式选择”，但并没有明显降低真实目录场景下的总体操作量
  - 单文件链路里的语言识别、方向提示、输出命名与执行稳定性已经较成熟，目录批量更适合建立在这条链路之上，而不是再维护一套 GUI 特化语义
  - 目录里常常会混有历史生成文件，如果不增加保护，批量模式很容易把 `_en` / `_zh` 输出再次当作输入翻译
- 影响范围：
  - 主要影响 GUI 主入口、目录批量确认交互、批量任务摘要展示与用户对“翻译方向”控件的理解
  - 不影响服务层保留目录扫描型 batch 能力，也不影响 benchmark / 兼容场景继续使用目录批量服务
- 处理：
  - 将 GUI 主入口从“单文件 / 多文件”收回为“单文件 / 目录批量”，取消多文件作为主流程
  - 在目录批量中复用单文件已有的文档语言识别能力，对每个文件分别自动判断 `zh_to_en` / `en_to_zh`
  - 将 GUI 中的方向单选语义改为“默认翻译方向”，仅对未确定语言的文件生效，并补充说明文案
  - 在开始前增加目录批量确认弹窗，展示中文 / 英文 / 未确定文件数量、默认兜底方向、跳过的不支持文件数与自动跳过的疑似生成输出数
  - 在批量执行计划中增加对历史生成输出的保护：若同目录存在可推断的原始源文件，则自动跳过 `*_en` / `_zh` 这类疑似生成结果
  - 在任务摘要和运行日志中追加批量计划摘要，便于用户在真正开始后仍能回看这次目录批量的识别分布与兜底策略
  - 同步更新 README、使用指南和相关 GUI / service 回归测试，并重新打包 Windows 发布产物
- 当前状态：已完成
- 补充结论：
  - 这次调整的核心不是“减少功能”，而是把 GUI 主路径重新收敛到更稳定、心智更统一的目录批量方案
  - 批量方向识别优先复用单文件成熟能力，只有未确定文件才退回用户选定的默认方向，这样既保留自动化，又避免黑盒感过强

### 问题 055：任务摘要缺少时间信息，且 GUI 没有直接入口查看当前任务完整日志，影响试用与排障效率

- 发现阶段：目录批量主流程稳定后，结合用户对可观察性的后续反馈
- 现象：
  - 主窗口“任务摘要”虽然已经从逐条细日志收敛为高信号摘要，但每一条摘要没有时间，用户难以快速判断最近状态刷新发生在什么时候
  - 详细运行日志虽然已经按任务写入 `logs/`，但 GUI 内没有直接入口查看，用户需要手动到输出目录找日志文件
  - 对于长任务或失败复盘，用户既希望主界面保持简洁，也希望在需要时立刻查看当前任务完整日志
- 原因：
  - 之前的摘要区更偏向“状态列表”，没有把时间维度纳入主界面可视信息
  - 日志落盘逻辑已经存在，但缺少一个轻量、就地的 GUI 访问入口
- 影响范围：
  - 主要影响 GUI 的可观察性、长任务试用体验和失败排障效率
  - 不改变底层 pipeline / batch service 的日志生产方式，也不改变现有日志文件格式
- 处理：
  - 统一收口任务摘要的追加入口，为所有摘要行增加 `[HH:MM:SS]` 时间前缀
  - 在主窗口操作区新增“查看详细日志”按钮；只有当前任务日志路径已建立时才启用，避免无效点击
  - 点击后以只读弹窗直接展示当前任务完整日志内容，不依赖外部编辑器或系统 shell
  - 若日志文件缺失或读取失败，GUI 给出明确告警并带出日志路径
  - 补充主窗口回归测试，覆盖摘要时间戳、日志按钮启用条件、日志查看成功路径与读取失败提示
  - 基于上述改动重新跑通定向 GUI 测试与全量 pytest，并重打 Windows 发布包
- 当前状态：已完成
- 补充结论：
  - 这次优化的重点是“让摘要承担轻量时间线，让日志承担完整上下文”，两者职责更清晰
  - 对用户来说，主界面仍保持简洁，但从试用到排障都比之前少了一步手动找日志文件的成本

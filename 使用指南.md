# 文档翻译工具使用指南

当前发布版本：`v0.4.1`

## 1. 这是什么

这是一个用于翻译单个 Markdown 文档的桌面工具。

支持：

- `中译英`
- `英译中`
- 自动识别常见中英文文档并切换推荐翻译方向
- 保留 Markdown 结构
- 输出新文件，不覆盖原文件

适用输入：

- 单个 `.md` 文件

## 2. 最快使用方法

1. 解压整个 `DocTranslationTool-win64.zip`
2. 打开 `dist\DocTranslationTool\`
3. 确认同目录下有 `DocTranslationTool.exe` 和 `.env`
4. 打开 `.env`，只先改这 3 项：

```env
DOC_TRANS_BASE_URL=你的接口地址
DOC_TRANS_API_KEY=你的API Key
DOC_TRANS_MODEL=你要用的模型名
```

5. 保存 `.env`
6. 双击运行 `DocTranslationTool.exe`
7. 先看左下角“模型状态”
8. 如果显示“配置不完整”，先回去补齐 `.env` 的 3 个关键项
9. 选择一个 `.md` 文件和输出目录
10. 工具会按文件内容自动切换推荐方向，你也可以手动改
11. 点击“开始翻译”

注意：

- 不要直接在 zip 压缩包内运行
- 不要只单独拷贝 `exe`，应保留整个目录结构

## 3. 打包版目录

主程序目录：

```text
dist\DocTranslationTool\
```

主程序：

```text
dist\DocTranslationTool\DocTranslationTool.exe
```

## 4. 用户只需要改什么

程序启动后会从 `DocTranslationTool.exe` 同目录读取这些文件：

- `.env`
- `settings.json`
- `glossary.json`

目录里同时还会附带一些模板或说明文件：

- `settings.example.json`
- `glossary.example.json`
- `CHANGELOG.md`
- `使用指南.md`

最重要的是：

- 直接改 `.env`

程序启动后，左下角会先给出一个大概状态：

- `模型状态：配置不完整`：`base_url`、`api_key`、`model` 至少缺一项
- `模型状态：配置已加载，待检查连通性`：关键项已读到，但还没真正发请求
- `模型状态：配置加载失败`：配置文件本身可能格式有问题

对大多数用户来说：

- 只需要改 `.env`
- 不需要改 `settings.json`
- 不需要改代码
- 不需要重新打包

最常改的只有 3 项：

- `DOC_TRANS_BASE_URL`：接口地址
- `DOC_TRANS_API_KEY`：接口密钥
- `DOC_TRANS_MODEL`：模型名

## 5. 最常改的配置

当前工具默认使用 `.env` 作为主要运行配置。

一个典型配置如下：

```env
DOC_TRANS_PROVIDER=openai_compatible
DOC_TRANS_API_FORMAT=openai
DOC_TRANS_ANTHROPIC_VERSION=2023-06-01
DOC_TRANS_BASE_URL=https://your-service.example/v1
DOC_TRANS_API_KEY=your_api_key_here
DOC_TRANS_MODEL=your_model_name
DOC_TRANS_TIMEOUT=60
DOC_TRANS_CONNECT_TIMEOUT=10
DOC_TRANS_READ_TIMEOUT=60
DOC_TRANS_MAX_RETRIES=2
DOC_TRANS_BATCH_SIZE=8
DOC_TRANS_PARALLEL_BATCHES=2
DOC_TRANS_TEMPERATURE=0.2
DOC_TRANS_FRONT_MATTER_FIELDS=title,subtitle,desc
```

重点字段说明：

- `DOC_TRANS_BASE_URL`：接口地址
- `DOC_TRANS_API_KEY`：接口密钥
- `DOC_TRANS_MODEL`：模型名
- `DOC_TRANS_API_FORMAT`：接口格式，可填 `openai` 或 `anthropic`
- `DOC_TRANS_ANTHROPIC_VERSION`：当接口格式为 `anthropic` 时使用的协议版本头
- `DOC_TRANS_BATCH_SIZE`：单批次片段数
- `DOC_TRANS_PARALLEL_BATCHES`：允许同时发送的批次数
- `DOC_TRANS_FRONT_MATTER_FIELDS`：允许翻译的 front matter 字段

如果你不确定怎么填：

- 先只改 `DOC_TRANS_BASE_URL`
- 再改 `DOC_TRANS_API_KEY`
- 最后改 `DOC_TRANS_MODEL`
- 其他保持默认即可

和速度最相关的通常是这两项：

- `DOC_TRANS_BATCH_SIZE`：每次请求里装多少片段
- `DOC_TRANS_PARALLEL_BATCHES`：同时跑多少个批次

如果大文档很慢，可以优先观察这两项，但不要一次调太大，否则更容易触发限流或结果不稳定。

## 6. 如果要切换模型或 Key

直接修改同目录下的 `.env` 即可。

最常见的是改这三项：

```env
DOC_TRANS_BASE_URL=https://your-service.example/v1
DOC_TRANS_API_KEY=your_api_key_here
DOC_TRANS_MODEL=your_model_name
```

如果你们平台支持其他模型，也可以直接替换 `DOC_TRANS_MODEL`。

如果你们平台同时提供两种兼容格式：

- OpenAI 兼容格式：

```env
DOC_TRANS_PROVIDER=openai_compatible
DOC_TRANS_API_FORMAT=openai
```

- Anthropic 兼容格式：

```env
DOC_TRANS_PROVIDER=openai_compatible
DOC_TRANS_API_FORMAT=anthropic
DOC_TRANS_ANTHROPIC_VERSION=2023-06-01
```

## 7. 如果只想演示流程，不想占用真实额度

可以临时改成离线演示模式：

```env
DOC_TRANS_PROVIDER=mock
DOC_TRANS_MODEL=mock-demo
```

说明：

- `mock` 模式不会调用外部 API
- 可以完整验证 GUI、解析、分段、回填和输出流程
- 输出结果是模拟翻译内容，只用于演示，不用于正式产出

演示结束后，记得切回真实配置：

```env
DOC_TRANS_PROVIDER=openai_compatible
```

## 8. 正常使用步骤

1. 双击打开 `DocTranslationTool.exe`
2. 先确认左下角模型状态不是“配置不完整”
3. 在“目标翻译文件”里选择一个 `.md` 文件
4. 工具会自动尝试判断文件是中文还是英文，并切换推荐翻译方向
5. 在“生成目录”里选择输出目录
6. 如有需要，再手动改翻译方向
7. 点击“开始翻译”
8. 开始后左下角会从“待检查连通性”进入“正在检查接口连通性”
9. 检查通过后进入翻译阶段，进度条会按批次推进
10. 到输出目录查看生成的新文件

第一次使用建议：

1. 先找一个较小的 `.md` 文件测试
2. 先确认能正常连接模型
3. 先确认能生成输出文件
4. 确认结果正常后，再翻译大文档

输出命名规则：

- 中译英：`原文件名_en.md`
- 英译中：`原文件名_zh.md`

翻译过程里你通常会看到这些状态：

- `翻译中：正在处理第 x/N 批`：当前至少有一个批次正在执行
- `翻译中：已完成批次 x/N`：已经有批次返回结果，进度条会继续向前推进
- 完成后会显示总耗时，日志里还能看到总批次数和重试次数

## 9. 推荐首次测试方式

第一次建议先用一个较小的 Markdown 文件测试，不要直接上超大文档。

首次测试建议先确认：

- 能正常打开 GUI
- 能通过模型连通性检查
- 能生成输出文件
- 输出 Markdown 结构没有被破坏

确认这些都正常后，再交给更大的真实文档使用。

## 10. 术语表怎么用

如果需要控制专有名词翻译，可以在 exe 同目录放一个 `glossary.json`。

示例：

```json
[
  {
    "source": "bootpackage",
    "target": "bootpackage"
  },
  {
    "source": "U-Boot",
    "target": "U-Boot"
  }
]
```

字段说明：

- `source`：原文术语
- `target`：希望输出时使用的术语

## 11. Front Matter 规则

如果 Markdown 文件开头带 YAML front matter：

- 会保留原有结构
- key 不会被翻译
- 默认只翻这些字段值：
  - `title`
  - `subtitle`
  - `desc`

默认不翻这些字段值：

- `author`
- `date`
- `ver`

如果要改可翻字段，可修改：

```env
DOC_TRANS_FRONT_MATTER_FIELDS=title,subtitle,desc
```

## 12. 常见问题

### 12.1 双击后打不开

先检查：

- 当前目录中是否有 `DocTranslationTool.exe`
- 是否被安全软件拦截
- 是否是在完整的 `dist\DocTranslationTool\` 目录下运行
- 是否是从 zip 里直接双击启动

### 12.2 模型连接失败

重点检查：

- `DOC_TRANS_BASE_URL` 是否正确
- `DOC_TRANS_API_KEY` 是否正确
- `DOC_TRANS_MODEL` 是否存在
- 网络或代理是否干扰连接

最常见原因就是这 3 项没有填对：

- 接口地址写错
- API key 失效
- 模型名在平台上不存在

如果错误里明确出现 `429 TOO MANY REQUESTS`：

- 这通常不是配置写错
- 更像是服务端限流、并发过高，或平台配额限制
- 可以稍后重试，或更换有额度的 key

如果错误里明确出现 `week allocated quota exceeded`：

- 说明当前 key 或接口通道的周额度已经用完
- 需要更换 key，或联系接口管理员确认重置时间

### 12.3 左下角模型状态怎么看

最常见的几种含义如下：

- `模型状态：配置不完整`：先检查 `.env` 的 `DOC_TRANS_BASE_URL`、`DOC_TRANS_API_KEY`、`DOC_TRANS_MODEL`
- `模型状态：配置已加载，待检查连通性`：本地配置看起来没问题，但还没真正访问接口
- `模型状态：正在检查接口连通性`：程序已经开始发起模型检查请求
- `模型状态：接口连通 (...)`：接口检查通过，可以继续执行翻译
- `模型状态：接口检查失败`：配置读到了，但请求没通过，重点查网络、代理、地址或 key

### 12.4 为什么进度条会按批次变化

现在进度条不是只看“有没有开始翻译”，而是会结合批次状态推进。

你会看到：

- `正在处理第 x/N 批`
- `已完成批次 x/N`

这比一直停在一个固定百分比更接近真实进度，尤其是大文档或并行批次较多时更容易判断程序有没有卡住。

### 12.5 术语表加载失败

重点检查：

- `glossary.json` 是否是 UTF-8 编码
- 顶层是否是 JSON 数组
- 每一项是否都有 `source` 和 `target`

### 12.6 输出失败

重点检查：

- 输出目录是否存在
- 输出目录是否有写权限
- 目标文件是否被其他程序占用

### 12.7 Windows 提示潜在风险或潜在勒索程序

这类提示通常不是因为程序逻辑异常，而是 Windows 对未签名打包程序的安全拦截或受控文件夹访问提醒。

建议：

- 先确认 exe 来源是公司内部交付版本
- 先用普通测试目录做输出，不要一开始就写桌面、文档等重点保护目录
- 如需长期分发，建议后续补代码签名
